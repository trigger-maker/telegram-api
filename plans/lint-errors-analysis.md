# Анализ ошибок линтера

**Дата анализа:** 2026-03-19  
**Проект:** telegram-api  
**Всего проблем (по данным пользователя):** 348 (288 ошибок, 60 предупреждений)

> **Примечание:** Поскольку Go и Node.js не установлены в текущей среде, линтеры не могут быть запущены напрямую. Этот анализ основан на конфигурациях линтеров и статическом анализе кода.

## 1. Go Backend (golangci-lint)

### Конфигурация линтера

**Файл конфигурации:** [`.golangci.yml`](../.golangci.yml)

**Включенные линтеры:**
- `gocyclo` - цикломатическая сложность
- `govet` - статический анализ Go кода
- `staticcheck` - проверка на ошибки, стиль и производительность
- `errcheck` - проверка обработанных ошибок
- `unused` - поиск неиспользуемого кода
- `gocritic` - дополнительные проверки качества кода
- `revive` - быстрый линтер для Go
- `dupl` - поиск дублированного кода
- `goconst` - поиск констант, которые должны быть вынесены
- `misspell` - проверка опечаток
- `lll` - проверка длины строк (максимум 140 символов)
- `funlen` - проверка длины функций (максимум 100 строк, 50 операторов)
- `gocognit` - когнитивная сложность (максимум 25)
- `godot` - проверка комментариев на наличие точки в конце
- `gosec` - проверка безопасности

### Предполагаемые категории ошибок

#### 1.1. Длина строк (lll) - ~20-30 ошибок

**Лимит:** 140 символов

**Типичные места возникновения:**
- Длинные SQL запросы в файлах миграций
- Длинные сообщения логов с эмодзи
- Длинные строки импорта

**Примеры из кода:**
```go
// internal/handler/session_handler.go:56
logger.Error().Err(err).Msg("❌ Error parseando body en Create session")

// internal/handler/session_handler.go:61
logger.Warn().Interface("errors", errs).Msg("⚠️ Validación fallida en Create session")
```

**Рекомендация:** Разбивать длинные строки на несколько строк.

---

#### 1.2. Цикломатическая сложность (gocyclo, gocognit) - ~30-40 ошибок

**Лимиты:**
- gocyclo: 15
- gocognit: 25

**Типичные места возникновения:**
- Обработчики HTTP запросов с множеством проверок
- Функции обработки ошибок
- Функции инициализации с множеством опций

**Предполагаемые файлы:**
- `internal/handler/session_handler.go`
- `internal/handler/chat_handler.go`
- `internal/service/chat_service.go`
- `internal/telegram/manager.go`

**Примеры из кода:**
```go
// internal/handler/session_handler.go:48-91
func (h *SessionHandler) Create(c *fiber.Ctx) error {
    // Множество проверок и условий
    userID, err := middleware.GetUserID(c)
    // ...
    if err := c.BodyParser(&req); err != nil {
        // ...
    }
    if errs := ValidateStruct(&req); errs != nil {
        // ...
    }
    // ...
}
```

**Рекомендация:** Выносить сложную логику в отдельные функции.

---

#### 1.3. Длина функций (funlen) - ~20-30 ошибок

**Лимиты:** 100 строк, 50 операторов

**Типичные места возникновения:**
- Обработчики HTTP запросов
- Функции инициализации
- Функции с множеством проверок

**Предполагаемые файлы:**
- `internal/handler/session_handler.go`
- `internal/handler/chat_handler.go`
- `internal/service/chat_service.go`

**Рекомендация:** Разбивать длинные функции на более мелкие.

---

#### 1.4. Необработанные ошибки (errcheck) - ~40-50 ошибок

**Типичные места возникновения:**
- Закрытие ресурсов
- Запись в логи
- Операции с базой данных

**Примеры из кода:**
```go
// internal/telegram/manager.go
func (m *ClientManager) SetPool(pool *SessionPool) {
    m.mu.Lock()
    defer m.mu.Unlock()  // errcheck может предупреждать здесь
    m.pool = pool
}
```

**Рекомендация:** Проверять ошибки или явно игнорировать с `_`.

---

#### 1.5. Дублированный код (dupl) - ~15-20 ошибок

**Порог:** 150 токенов

**Типичные места возникновения:**
- Похожие обработчики ошибок
- Похожие валидации
- Похожие операции CRUD

**Предполагаемые файлы:**
- `internal/handler/*_handler.go` - похожие паттерны обработки
- `internal/service/*_service.go` - похожие CRUD операции

**Рекомендация:** Выносить общий код в отдельные функции или использовать шаблоны.

---

#### 1.6. Комментарии без точки (godot) - ~20-30 ошибок

**Типичные места возникновения:**
- Doc-комментарии для экспортируемых функций
- Комментарии к структурам

**Примеры из кода:**
```go
// internal/telegram/manager.go:16
// TGUser represents a Telegram user  // Нет точки в конце

// internal/telegram/manager.go:32
// QRAuthResult represents the result of QR authentication  // Нет точки в конце
```

**Рекомендация:** Добавлять точку в конце комментариев.

---

#### 1.7. Неиспользуемый код (unused) - ~10-15 ошибок

**Типичные места возникновения:**
- Неиспользуемые импорты
- Неиспользуемые переменные
- Неиспользуемые функции

**Рекомендация:** Удалять неиспользуемый код.

---

#### 1.8. Проблемы безопасности (gosec) - ~15-20 ошибок

**Типичные места возникновения:**
- Использование `math/rand` вместо `crypto/rand`
- Небезопасное использование `eval`
- Проблемы с SQL инъекциями

**Рекомендация:** Использовать безопасные альтернативы.

---

#### 1.9. Проблемы стиля (revive, gocritic) - ~30-40 ошибок

**Типичные места возникновения:**
- Именование переменных
- Стиль кода
- Неэффективные паттерны

**Рекомендация:** Следовать рекомендациям линтера.

## 2. TypeScript Frontend (ESLint)

### Конфигурация линтера

**Файл конфигурации:** [`frontend/eslint.config.js`](../frontend/eslint.config.js)

**Включенные правила:**

#### TypeScript правила:
- `@typescript-eslint/no-unused-vars` - неиспользуемые переменные
- `@typescript-eslint/no-explicit-any` - использование `any` типа

#### React правила:
- `react-hooks/rules-of-hooks` - правила хуков
- `react-hooks/exhaustive-deps` - зависимости хуков

#### Правила качества кода:
- `max-len` - максимальная длина строки (120 символов)
- `max-lines-per-function` - максимальное количество строк в функции (50)
- `complexity` - сложность кода (6)
- `max-depth` - максимальная глубина вложенности (4)
- `max-params` - максимальное количество параметров (4)
- `no-console` - использование console
- `prefer-const` - предпочтение const
- `no-var` - запрет var

### Предполагаемые категории ошибок

#### 2.1. Неиспользуемые переменные (@typescript-eslint/no-unused-vars) - ~20-30 ошибок

**Типичные места возникновения:**
- Деструктуризация с неиспользуемыми полями
- Импорты, которые не используются
- Параметры функций

**Примеры из кода:**
```typescript
// frontend/src/hooks/useSessions.ts:41
export const useVerifyCode = () => {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: ({ sessionId, code }: { sessionId: string; code: string }) =>
      sessionsApi.verifyCode(sessionId, { code }),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: [SESSIONS_QUERY_KEY] })
    },
  })
}
// queryClient может быть не использован в некоторых хуках
```

**Рекомендация:** Удалять неиспользуемые переменные или использовать префикс `_`.

---

#### 2.2. Длина строк (max-len) - ~15-25 ошибок

**Лимит:** 120 символов

**Типичные места возникновения:**
- Длинные импорты

// frontend/src/api/client.ts:48
throw new ApiException(

#### 2.7. Использование any (@typescript-eslint/no-explicit-any) - ~10-15 ошибок

**Типичные места возникновения:**
- API запросы
- Обработка ошибок

**Примеры из кода:**
```typescript

public async post<T>(url: string, data?: any): Promise<T> {
  const response = await this.client.post<ApiResponse<T>>(url, data)
  return response.data.data as T
}
```

**Рекомендация:** Использовать типы вместо `any`.

---

#### 2.8. Зависимости хуков (react-hooks/exhaustive-deps) - ~10-20 ошибок

**Типичные места возникновения:**
- useEffect с неполными зависимостями
- useCallback с неполными зависимостями

**Примеры из кода:**
```typescript
// frontend/src/components/sessions/QRCodeModal.tsx:32-49
useEffect(() => {
  if (!isOpen) return

  const interval = setInterval(async () => {
    const result = await refetch()
    // ...
  }, 3000)

  return () => clearInterval(interval)
}, [isOpen, sessionId, refetch, onSuccess, onClose])
// refetch может меняться при каждом рендере
```

**Рекомендация:** Добавлять все зависимости или использовать useCallback.

## 4. Приоритеты исправления

### Высокий приоритет (влияют на функциональность или безопасность)
1. **Необработанные ошибки (errcheck)** - могут привести к пропущенным ошибкам
2. **Проблемы безопасности (gosec)** - уязвимости безопасности
3. **Зависимости хуков (react-hooks/exhaustive-deps)** - могут привести к багам в React

### Средний приоритет (влияют на качество кода)
1. **Цикломатическая сложность (gocyclo, gocognit)** - сложность поддержки
2. **Длина функций (funlen, max-lines-per-function)** - сложность поддержки
3. **Дублированный код (dupl)** - сложность поддержки
4. **Сложность кода (complexity)** - сложность поддержки

### Низкий приоритет (стилистические проблемы)
1. **Длина строк (lll, max-len)** - форматирование
2. **Комментарии без точки (godot)** - документация
3. **Проблемы стиля (revive, gocritic)** - стиль кода
4. **Использование console (no-console)** - отладка

---

## 5. Рекомендации по исправлению

### Для Go Backend:
1. Создать утилиты для обработки ошибок (error wrapping)
2. Вынести общую логику в отдельные функции
3. Использовать интерфейсы для уменьшения дублирования
4. Добавить точки в конец комментариев
5. Использовать `crypto/rand` вместо `math/rand`

### Для TypeScript Frontend:
1. Использовать типы вместо `any`
2. Выносить логику в отдельные хуки
3. Использовать `useCallback` для стабилизации зависимостей
4. Разбивать длинные компоненты на более мелкие
5. Использовать систему логирования вместо console

## 6. Следующие шаги

Для точного определения ошибок необходимо:

1. **Установить Go и Node.js** в среду разработки
2. **Запустить golangci-lint:**
   ```bash
   golangci-lint run --out-format=json > golangci-report.json
   ```
3. **Запустить ESLint:**
   ```bash
   cd frontend
   npx eslint . --format json > eslint-report.json
   ```
4. **Проанализировать JSON отчеты** и создать конкретные подзадачи для исправления
